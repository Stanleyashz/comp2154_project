Goal: Move system from "Mostly working" → "Consistently working."

**Types of Testing to Perform:**
1. **Functional testing** — Does the feature work?
2. **Integration testing** — Do components work together?
3. **Workflow testing** — Can users complete tasks end-to-end?
4. **Edge case testing** — What happens with unexpected input?

**Executing Test Cases — Steps:**
1. Review planned test cases
2. Execute each test step
3. Record expected vs actual results
4. Document failures or unexpected behavior
- Testing must be **structured and repeatable**

**Example Test Case Format (TC-AUTH-001 - Successful User Login):**
```
Test Case ID:        TC-AUTH-001
Test Case Name:      Successful User Login
Requirement Ref:     FR-1 – User Authentication
Priority:            High
Test Level:          Functional / System
Test Type:           Positive
Pre-Conditions:      User account exists, account is active,
                     services operational, user not currently logged in

Test Steps:
1. Launch the web application in a supported browser
2. Navigate to the Login page (/login)
3. Enter valid email and password
4. Click Login

Expected Results:
- User is authenticated successfully
- User is redirected to the dashboard
- Authenticated session is created
- No error messages are displayed
```

**What is a Defect?**
A defect occurs when the system does **not behave as expected**.

Examples:
- Application crashes
- Data not saved correctly
- Incorrect results returned
- UI displays incorrect information
- Security or validation failures

> Defects must be **logged and tracked**, not ignored.

**Defect Log Fields (per defect):**
- Defect title
- Description of the problem
- Steps to reproduce
- Expected behavior
- Actual behavior
- Priority level
- Assigned team member

**Recommended Defect Tracking Tools:**
- GitLab Issues
- GitHub Issues
- Jira
- Trello
- Spreadsheet defect tracker

**Defect Priority Levels:**

| Priority | Definition |
|----------|-----------|
| 🔴 Critical | System cannot function |
| 🟠 Major | Important feature fails |
| 🟡 Minor | Small issues or cosmetic bugs |

> Always fix **Critical defects first**.

**Defect Resolution Process:**
1. Identify defect
2. Reproduce issue
3. Investigate cause
4. Implement fix
5. Re-test feature
6. Confirm resolution
- After fixing a defect, **always run the test again**

**Improving System Quality (beyond bug fixes):**
- Improving validation
- Improving error messages
- Refactoring complex code
- Improving documentation
- Improving user feedback

**Evidence to Include in Progress Report 3:**

    - What remains before final delivery

**Sample Testing Table (CEMS example for reference):**
```
Test Category          | # Cases | Passed | Failed | Notes
-----------------------|---------|--------|--------|---------------------------
Authentication         |   12    |   11   |    1   | Invalid token edge case fixed
Event Creation / CRUD  |   15    |   14   |    1   | Optional field validation fixed
Registration/Cancel    |   10    |    9   |    1   | Registration count bug fixed
Role-Based Access      |    8    |    7   |    1   | Student route leak patched
Attendance Tracking    |    6    |    6   |    0   | All tests passed
TOTAL                  |   51    |   47   |    4   | ~92% pass rate after fixes
```

**Sample Defect Log (CEMS example for reference):**
```
DEF-001 | Optional field validation error       | Medium | Resolved | Week 7
DEF-002 | Students accessing organizer routes   | High   | Resolved | Week 8
DEF-003 | Registration count not decrementing   | High   | Resolved | Week 8

```

**Key Lesson from Sample:**
> "By executing system tests while features were still being developed, the team identified defects early and avoided larger issues later. Implementing automated integration tests earlier in the development lifecycle reduces manual testing effort during later phases."
